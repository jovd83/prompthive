# Refactoring Roadmap: prompt-hive

## Executive Summary
**Maintainability Score: 55/100**

**State of the Union:** 
Prompt-hive operates on a modern, robust stack (Next.js 16 App Router, Prisma, Tailwind) but suffers from "prototype hangover" characteristics such as tightly coupled architectural layers, rampant silent failures, and eroded type boundaries. To mature into a production-grade application, the primary focus must be extracting data access logic out of UI Server Components, breaking down monolithic service files (`services/prompts.ts`), and enforcing strict error handling and type safety protocols.

## The Audit Table

| Location | Severity | Smell Detected | The Risk | The Fix |
| :--- | :--- | :--- | :--- | :--- |
| `app/(dashboard)/**/page.tsx` | High | Boundary Violations (Data in UI Layer) | Direct `prisma` calls in Next.js Server Components create tight coupling. Features cannot be tested in isolation without spinning up a database. | Introduce a dedicated Data Access Layer (DAL) or utilize existing `services/` uniformly. UI components should only call asynchronous service functions. |
| `app/(dashboard)/settings/page.tsx:72` | Med | Leaky Abstractions (Raw DB Query) | `prisma.$queryRaw<any[]>\`SELECT * FROM "GlobalConfiguration"\`` bypasses the ORM's type safety and schema definitions, opening the door to runtime errors on schema evolution. | Replace with standard Prisma `findUnique` calling the model safely, avoiding raw SQL strings. |
| `services/prompts.ts` | High | God Object / Side-Effect Soup | An 800+ line monolith combining complex DB transactions, filesystem I/O (`fs`, `path`), attachment parsing, and visual features (`generateColorFromName`). | Split into distinct domain services: `PromptCrudService`, `PromptAttachmentService`, and `TagManagementService`. |
| Widespread (`vitest.setup.ts`, `services/user.ts`, etc.) | Med | Type Safety Erosion (`as any`) | Pervasive `as any` casting defeats TypeScript. It provides false confidence and masks nested type mismatches during refactors or schema changes. | Introduce strict Zod schemas for validation, replace `any` with `unknown` where dynamic, and properly type mocked dependencies using `ReturnType` or standard factories. |
| `services/backup.ts`, `services/prompts.ts` | Critical | The "Silent Failure" Trap (`catch (e) {}`) | Empty catch blocks suppress critical operational failures (I/O, DB constraints). In-production state will mutate unpredictably and block debugging efforts. | Implement a centralized structured logger (e.g., Winston, Pino) and replace empty blocks with `.catch()` handlers that log error context and intentionally re-throw or return generic failure states. |
| Widespread Layouts & Queries | Low | Magic Values | Hardcoded strings like `"GLOBAL"` for configuration IDs, and `"USER"`/`"ADMIN"` for roles appear throughout the business logic. | Create a `src/constants/` module exporting Enums (e.g., `export const CONFIG_ID = { GLOBAL: 'GLOBAL' }`) and use them strictly throughout the system. |

# Refactoring Roadmap: prompt-hive (Round 4)

## Executive Summary
**Maintainability Score: 87/100**

**State of the Union:** 
Significant progress has been made across the last 3 phases. The boundary between the presentation and data layers is tightly regulated, and UI components no longer crash silently when network failures occur. The God Component `CollectionSidebar` has been decomposed into custom hooks. 
However, this codebase still relies on TypeScript compiler "bypasses" as a crutch to pass deeply nested objects from Server Components into Client Components. Additionally, the Server Actions hub (`actions/prompts.ts`) is accumulating too much diverse logic, violating the Single Responsibility Principle.

## The Audit Table

| Location | Severity | Smell Detected | The Risk | The Fix |
| :--- | :--- | :--- | :--- | :--- |
| `actions/prompts.ts` (~12.5 KB) | High | Action Module Bloat | Currently manages 15+ completely separate domain behaviors: Standard CRUD operations, Drag-and-drop Bulk Mutations, Cross-referencing logic (Links), and Version Control. Merge conflicts and poor discoverability are guaranteed as the app scales. | Create a modular action namespace. Partition this file into `actions/prompt-crud.ts`, `actions/prompt-bulk.ts` and `actions/prompt-links.ts`. |
| `app/(dashboard)/**/*.tsx`<br>and `components/*.tsx` | High | Type Safety Erosion (`as any`) | Deeply populated Prisma calls (e.g., fetching a prompt with all its versions, authors, and favorites) construct giant TS Interfaces. Because the Client Components expect highly specific formats, the `as any` sledgehammer was used in over 20 files to bypass the compiler entirely. | Modify `types/prisma.ts` to fully encapsulate the schema augmentations (e.g., adding `isPrivate` and `isLocked` to `PromptWithRelations`). Remove the explicit runtime augmentations and `as any` casts in `PromptDetail.tsx`, `TagList.tsx`, and `app/(dashboard)/layout.tsx`. |
| `app/(dashboard)` Server Components | Medium | Data Leaks / Excessive Over-fetching | Server components are sending entire database trees to the client side because of loose Prisma validations. E.g. passing `collections as any` instead of strictly mapping to a DTO (Data Transfer Object) format. | Build explicit presentation validators/mappers that trim Prisma models down *before* sending them across the server/client boundary. |
| `components/ExportZeroForm.tsx`<br>`components/StandardExportForm.tsx` | Low | Magic Values / Hardcoded Slugs | There are still a few raw text strings and hard-coded fallback values scattered in the presentation layer logic. | Formalize these mappings or localize them in standard constants if they represent system mechanics. |

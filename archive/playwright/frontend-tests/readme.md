# Experiment: Microsoft Playwright Agents

This directory contains the results of an experiment with **Microsoft Playwright Agents**, as described in the [Playwright documentation](https://playwright.dev/docs/test-agents). The goal was to evaluate how native agentic capabilities can streamline the creation and maintenance of E2E tests for PromptHive.

## Overview
The goal of this experiment was to test the effectiveness of Microsoft's agentic approach to testing. Instead of writing rigid, selector-heavy scripts, we used high-level instructions and natural language prompts to allow the Playwright Agent to autonomously navigate the application and verify behaviors.

## What has been created
- **`plans/`**: Functional test plans generated by analyzing the application's Epics, acting as the "source of truth" for the Agent.
- **`*.spec.ts`**: Playwright test files that utilize the agentic patterns to interact with the UI.
- **`utils.ts`**: Support functions to handle non-agentic setup tasks like session management and database state.
- **`archive/`**: Earlier iterations of tests used to compare traditional Playwright scripting against the Agent-driven approach.

## What is tested
The suite validates core user journeys through the lens of an intelligent agent:
*   **User Lifecycle**: Registration, login, and profile accessibility.
*   **Prompt Engineering Flow**: Creating prompts, managing variables, and version control.
*   **Organizational Logic**: Hierarchical collection management and prompt assignment.
*   **Discovery Engine**: Multi-dimensional search filtering and global command palette interactions.

## How it was decided what to test
The testing scope was derived directly from the project's **Epic documents**. We provided these requirements to the Agent, allowing it to identify the critical paths and edge cases that required validation based on the described user and business goals.

## How it was decided what to assert
- **Semantic Assertions**: The Agent determines the correct state by analyzing the page structure and visible text, mapping it back to the natural language goals.
- **Observational Verification**: Assertions are often "goal-based" (e.g., "Verify the search results update to show only SEO prompts") rather than just checking for a specific CSS class.
- **Heuristic Validation**: The Agent uses internal heuristics to confirm that navigation succeeded and that the UI matches the expected functional state.

## How the plans are constructed
1.  **Requirement Digest**: The Agent ingests a requirement Epic (e.g., `Epic_Search_And_Discovery.md`).
2.  **Scenario Discovery**: It breaks down the Epic into discrete, testable scenarios stored in the `plans/` directory.
3.  **Instruction Synthesis**: For each scenario, the Agent defines the sequence of actions and the expected outcome in natural language instructions.
4.  **Execution Logic**: The Agent then executes these instructions, dynamically resolving locators and handling UI state changes on the fly.

## Agent vs. Gemini: Implementation Roles
- **Microsoft Playwright Agent**: The core engine responsible for the interactive automation. It handles the "How" of interacting with the browser based on intent.
- **Gemini (The LLM)**: Acts as the "Brain" powering the Agent. It interprets the Epics to generate plans and translates high-level test goals into the executable intents the Playwright Agent performs.

## Operations: Run, Plan, Debug, Self-heal
- **Running**: Execute the suite using the standard Playwright runner:
  ```bash
  npx playwright test
  ```
- **Planning**: Fully automated from requirements to structured plans.
- **Debugging**: Uses the Playwright Trace Viewer to inspect the internal "thoughts" and actions of the Agent during execution.
- **Self-healing**: Unlike traditional tests, these are natively self-healing. Because the Agent relies on intent and semantic understanding rather than brittle CSS/XPath selectors, it can automatically adapt to many UI changes without requiring code updates. If the visual layout changes but the functional meaning remains, the test continues to pass.
